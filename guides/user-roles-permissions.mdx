This guide explains the three user roles in Pramukh Men's Wear and what each role can do in the system.

## Role Overview

Pramukh Men's Wear uses a role-based access control system with three levels:

- **Owner**: Full system access, manages all branches and users
- **Manager**: Branch-level management and administrative tasks
- **Cashier**: Day-to-day sales and customer service operations

Roles can be assigned both globally (in the user profile) and per-branch (in branch assignments), giving you flexibility in managing multi-location operations.

## Branch Assignments

Users can be assigned to one or multiple branches, and their role can differ per branch.

### How Branch Assignment Works

<Steps>
  <Step title="User Profile Role">
    Every user has a default role set in their profile:
    
    - This is their base permission level
    - Set when the user account is created
    - Can be changed by owners
  </Step>

  <Step title="Branch-Specific Role">
    When a user is assigned to a branch, they get a role for that branch:
    
    - Can be different from their profile role
    - Allows flexibility (e.g., manager at one branch, cashier at another)
    - Overrides the profile role when working at that branch
  </Step>

  <Step title="Active Branch Context">
    When a user logs in:
    
    - They select which branch they're working at
    - The system applies their role for that branch
    - All actions are scoped to that branch
    - Owners see data from all branches
  </Step>
</Steps>

### Multi-Branch Scenarios

**Scenario 1: Manager at Multiple Locations**
```
User: Rahul
Profile role: Manager
Branch assignments:
  - Mumbai Store: Manager
  - Pune Store: Manager
  
Rahul can manage inventory and purchases at both locations,
switching between them as needed.
```

**Scenario 2: Different Roles at Different Branches**
```
User: Priya
Profile role: Cashier
Branch assignments:
  - Flagship Store: Manager
  - Mall Outlet: Cashier
  
Priya has manager permissions at the flagship store but
cashier permissions at the mall outlet.
```

**Scenario 3: Owner with Branch Access**
```
User: Amit (Owner)
Profile role: Owner
Branch assignments:
  - All branches (automatic)
  
Amit can access and manage all branches from a single login,
view cross-branch reports, and manage users globally.
```

## Assigning Roles

Only owners can assign users to branches and change roles.

<Steps>
  <Step title="Navigate to User Management">
    From the admin panel, go to **Users** or **Team**.
    
    You'll see a list of all users with their current roles and branch assignments.
  </Step>

  <Step title="Add User to Branch">
    Select a user and click **"Assign to branch"**.
    
    Choose:
    - The branch to assign them to
    - Their role at that branch
    - Effective date (optional)
  </Step>

  <Step title="Change User Role">
    To change a user's role at a branch:
    
    1. Find the branch assignment
    2. Click **"Edit role"**
    3. Select the new role
    4. Save changes
    
    The change takes effect immediately.
  </Step>

  <Step title="Remove Branch Access">
    To remove a user from a branch:
    
    1. Find the branch assignment
    2. Click **"Remove from branch"**
    3. Confirm the removal
    
    The user loses access to that branch but retains access to other assigned branches.
  </Step>
</Steps>

## Security Best Practices

<AccordionGroup>
  <Accordion title="Follow the Principle of Least Privilege">
    Give users only the access they need:
    
    - New hires start as cashiers
    - Promote to manager only when they need inventory/purchase management
    - Keep owner role limited to actual business owners
    
    This reduces the risk of accidental changes or data exposure.
  </Accordion>

  <Accordion title="Review Access Regularly">
    Periodically audit user roles:
    
    - Remove access for users who leave
    - Downgrade roles when job responsibilities change
    - Verify branch assignments match current work locations
    
    Set a quarterly review schedule.
  </Accordion>

  <Accordion title="Document Role Changes">
    Keep notes on why roles were assigned or changed:
    
    - Promotion dates
    - Business justification
    - Who authorized the change
    
    This creates an audit trail for compliance and troubleshooting.
  </Accordion>

  <Accordion title="Train Users on Their Role">
    When assigning a role:
    
    - Explain what they can and cannot do
    - Show them relevant features
    - Document common tasks for their role
    - Provide a contact for permission requests
    
    Proper training prevents confusion and improper use.
  </Accordion>
</AccordionGroup>

<Accordion title="Can a cashier see other cashiers' sales?">
  It depends on system configuration:
  
  - By default, cashiers see all sales for their branch (for customer service)
  - Some businesses configure cashier-only views where they see only their own transactions
  - Managers and owners always see all sales
  
  Check with your owner or system administrator for your specific setup.
</Accordion>

<Accordion title="What happens if I'm assigned to multiple branches?">
  When logging in, you'll:
  
  1. See a branch selector on the home screen
  2. Choose which branch you're working at
  3. All your actions apply to that branch
  4. Your role for that branch determines your permissions
  5. Switch branches anytime from the branch selector
  
  This lets you manage multiple locations without separate accounts.
</Accordion>

<Accordion title="Can I temporarily elevate a cashier's permissions?">
  Not directly. To give a cashier manager permissions:
  
  1. Owner must edit their branch assignment
  2. Change role from Cashier to Manager
  3. The change takes effect immediately
  4. Change back when temporary need ends
  
  For one-time needs, consider having a manager perform the task instead.
</Accordion>

<Accordion title="Do role changes affect historical data?">
  No. Role changes only affect future actions:
  
  - Past sales show the original cashier
  - Old purchases retain the original recorder
  - Historical reports remain unchanged
  
  All historical data maintains its original context even if roles change.
</Accordion>
